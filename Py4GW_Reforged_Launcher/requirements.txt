# Dependency setup for the launcher's dev venv.
#
# Create the venv at .venv INSIDE this folder (Py4GW_Reforged_Launcher\.venv) --
# that exact location is the convention Launch_Reforged.bat depends on:
#     py -3.13-32 -m venv .venv
#
# Then install wheels-only (plain `pip install` falls back to source builds for
# some of these on 32-bit CPython 3.13 / Windows and fails without a C compiler),
# with one exception: pywebview's own dependency `proxy_tools` only ships an
# sdist, no wheel -- but it's pure Python (no C extension, no compiler needed),
# so it's safe to install from source:
#   .venv\Scripts\pip install --only-binary=:all: --no-binary=proxy_tools -r requirements.txt
#
# RELAY 043: trimmed after the old ImGui app (launcher.py, imgui-bundle) retired.
# Every package below was verified empirically, not guessed from grep alone --
# imgui-bundle uninstalled from the real dev venv, `pip check` run clean, and a
# real run_shell.py launch confirmed working end to end -- before removing
# annotated-types/glfw/munch/numpy/pillow/pydantic/pydantic_core/PyOpenGL/
# typing-inspection (all either exclusively required by imgui-bundle's own
# dependency tree, per `pip show <pkg>`'s real Required-by field, or -- typing-
# inspection specifically -- not even actually installed despite being pinned
# here). Every package kept below either has a real `import` somewhere in
# launcher_core/ or pywebview_shell/ (psutil, pywin32, dulwich, pywebview) or is
# a real, confirmed (via `pip show`) transitive dependency of one that is
# (bottle/proxy_tools/pythonnet <- pywebview, clr_loader <- pythonnet, cffi <-
# clr_loader, pycparser <- cffi, typing_extensions <- pywebview, urllib3 <-
# dulwich).
